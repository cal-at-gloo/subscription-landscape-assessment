Your Platform: Gloo Workspace

Understanding the Problem Space

Gloo's Workspace platform presents a uniquely layered subscription challenge: multiple standalone products under one roof, each with its own seat logic, free/paid tiers, and resource models. Two products — Content Studio and Ministry Chat — introduce AI-specific resource meters (credits vs. tokens), which is a pattern not widely solved cleanly in the market. The Gloo+ upgrade model (individual-level upgrade within an always-free product) is particularly rare and has few direct analogues.

Platform Breakdowns

Monday.com
Work OS · Multiple Products
Structural complexity

Subscription Architecture

Four standalone products (Work Management, CRM, Dev, Service), each with its own plan tiers and seat count. Products can be bundled but must share a billing cycle. Seat pricing uses "bucket" increments (5, 10, 15…), not per-seat exact pricing.

A Product Non-Member role exists for users subscribed to one product but not another — they get read-only access to the others automatically.

Per-product seats Bucket pricing Bundle model

Admin View & Seat Management

The Administration panel (accessible via avatar → Administration) hosts a Users tab with full member table: role, product access, department, seat status. Admins can bulk-edit roles, assign departments with reserved seat pools, and delegate department-level admin rights.

Custom roles are Enterprise-only. Viewers are free and unlimited across all plans.

Delegated dept. owners Viewer = free Guest tiering
Design Signal: Product Non-Member is a clever auto-downgrade — users keep read access to unsubscribed products without buying a seat. Pricing docs →

Google Workspace
Productivity Suite · Unified License
Structural complexity

Subscription Architecture

Flat per-user licensing — one edition (Starter, Business Starter, Business Standard, Business Plus, Enterprise) covers the full product suite. No per-product subscriptions. All Google apps are included; access is controlled at the org or OU level.

Unified license Org-level app toggles Edition-based features

Admin View & Seat Management

The Google Admin Console (admin.google.com) is a purpose-built, power-dense interface. Users are organized by Organizational Units (OUs) — the primary method for applying policies at scale. Admin roles are pre-built (User Management Admin, Groups Admin, etc.) or custom, with granular privilege scoping.

Super Admin role has exclusive access to billing, security, and account recovery. No concept of "seats" — purchasing is by edition per user.

Org Unit (OU) structure Pre-built + custom roles 750 custom role limit
Design Signal: OUs are a powerful but invisible structure — admins must learn the mental model. Great at scale; complex to onboard to. Admin roles docs →

Atlassian Cloud
Dev Tools Suite · Per-App Subscriptions
Structural complexity

Subscription Architecture

Each product (Jira Software, Confluence, Jira Service Management, Bitbucket, etc.) has its own Free / Standard / Premium / Enterprise plan, billed independently per user tier. Enterprise is billed at the org level across all instances.

Atlassian Guard (security layer) is a separate subscription. Free plans support up to 10 users; exceeding triggers auto-upgrade to Standard trial.

Per-app plans User tier billing Guard add-on

Admin View & Seat Management

admin.atlassian.com is the central org dashboard. Admins manage product access per user across all apps. Users are added at org level, then granted product-specific access. This is the closest analogue to a "hub" admin model.

Enterprise plan allows up to 150 instances per product type. Billing is complex: users billed per app, per tier. Adding users auto-counts toward billing even if they haven't accepted the invite yet.

Centralized org admin Auto-billing on invite Enterprise multi-instance
Design Signal: Billing on invite (before acceptance) is a UX friction point frequently cited by admins. Consider explicit "pending cost" indicators. User & billing docs →

Figma
Design Tool · Seat-Type Model (Evolved)
Structural complexity

Subscription Architecture

As of March 2025, Figma moved from per-product seats to a unified seat-type model: one seat per person, with the seat type determining product access. Seat types: Full (edit everything), Dev (dev tools + view), Collab (FigJam edit + Design view), View (free).

View seats are free and unlimited. AI credits are shared org-wide with pay-as-you-go option.

Unified single seat View = always free AI credits: org pool

Admin View & Seat Management

Admin panel lives in the file browser sidebar → Admin. People tab shows all members with seat type column — clickable to change. Seat request flow: users request an upgrade, admin gets notification, can approve/deny. Temporary 3-day access granted while pending.

Seat approval settings can be Auto-approve (if seat available) or Manual approval. On Enterprise, billing groups allow sub-org seat pools. New seats are prorated to next invoice.

Self-service seat requests 3-day temp access Prorated billing Billing groups (Ent.)
Design Signal: The seat request → temp access → approve/deny flow is a standout pattern. It lets users try before the admin commits cost — low friction for both sides. Seat management docs →

Slack
Messaging · Workspace / Org Hierarchy
Structural complexity

Subscription Architecture

Plans: Free, Pro, Business+, Enterprise Grid. Guest types (Single-Channel, Multi-Channel) are available on paid plans only and are counted differently from members. Enterprise Grid supports multiple workspaces within one org — a powerful hierarchical model.

Workspace hierarchy Guest tiering Enterprise Grid

Admin View & Seat Management

Workspace owners/admins access settings via Tools & Settings → Manage Members. Roles: Primary Owner → Owner → Admin → Full Member → Single/Multi-Channel Guest. System roles (e.g., Channels Admin, Security Admin, Workflow Admin) allow granular delegation without full admin access — an RBAC overlay on top of structural roles.

Enterprise allows custom roles. Org-level vs. workspace-level admin distinction is a key design pattern.

System roles (RBAC overlay) Org vs. workspace level Custom roles (Ent.)
Design Signal: Slack's system role overlay is elegant — it lets you give someone "Channels Admin" ability without making them a full workspace admin. Useful for Gloo's context. Roles docs →

Notion
Workspace · Member / Guest Model
Structural complexity

Subscription Architecture

Plans: Free, Plus ($10/mo), Business ($20/mo), Enterprise (custom). Seat = 1 member. Guests are always free but limited by plan (10 Free, 100 Plus, 250 Business). Mid-2025: bundled unlimited AI into Business and Enterprise — no more separate AI add-on.

1 seat = 1 member Guest = free AI bundled (Biz+)

Admin View & Seat Management

Settings → People is the member management hub. Roles: Organization Owner (Enterprise) → Workspace Owner → Membership Admin (Ent. only) → Member → Guest. The Membership Admin role is notable — it exists only on Enterprise and allows member management without workspace settings access.

When you remove a member, their seat stays "available" until billing period end — you can swap a new person in at no extra cost. This seat-recycling model is user-friendly for teams with turnover.

Seat recycling on removal Membership Admin role Suggestions: guests → members
Design Signal: Notion's "suggestions" feature (recommending guest-to-member upgrades) is a smart proactive upsell that feels helpful, not pushy. Billing docs →

Canva
Design Platform · Org → Team Hierarchy
Structural complexity

Subscription Architecture

Plans: Free, Pro ($15/mo), Business ($10/user/mo), Enterprise (custom). Canva Teams discontinued for new sign-ups; Business is the current team entry point. Enterprise adds multi-team structure, SSO, SCIM. Magic Studio AI features have usage caps (Free: 50 total; Pro/Business: 500/mo).

Business: 1 team max Enterprise: multi-team AI usage caps

Admin View & Seat Management

Org Admin → Team Admin hierarchy. Org Admins set default permissions for all teams; Team Admins can override if allowed. The Controls & Permissions section is layered: Org → Team tabs, with toggle-based settings per capability. Roles within teams: Member, Template Designer, Admin.

Brand Kits and publishing workflows have their own permission axis, separate from seat type — a parallel permission layer that's unique to Canva's use case.

Org default → team override Brand permissions axis Publishing approval flow
Design Signal: The "org sets defaults, teams can override" model is a powerful pattern for platforms with autonomous sub-groups sharing org-level brand standards. Roles & permissions docs →
☁️
Salesforce
CRM Platform · License × Profile × Permission
Structural complexity

Subscription Architecture

Three independent access layers: Edition (org-level feature ceiling: Essentials → Enterprise → Unlimited), User License (seat type per user: Salesforce, Platform, Identity, etc.), and Feature License (add-on entitlements: Marketing Cloud, Flow, etc.). Login-based licensing also available as alternative to seat-based.

3-layer access model Login-based option Feature licenses

Admin View & Seat Management

Setup → Users is the management hub. Access is controlled by: Profiles (baseline minimum access — every user must have one) + Permission Sets (additive access grants) + Permission Set Groups (bundles of permission sets). Salesforce is actively migrating away from profiles toward permission-set-led models.

The Role Hierarchy is a separate construct used for data visibility (record access), orthogonal to permission roles. The most complex access model of any platform reviewed.

Profiles → permission sets Role hierarchy = data visibility License Mgmt App
Design Signal: Salesforce's 3-layer model is powerful but notoriously complex — it's what admins complain about most. Avoid this in a B2SMB context; aim for 2 layers max. Admin guide →
🅰️
Adobe Creative Cloud
Creative Suite · Product Profile Model
Structural complexity

Subscription Architecture

Three plan types: Teams (direct purchase, per seat), VIP (volume contract), Enterprise (ETLA annual). Licenses are per-product or All Apps. A single All Apps license grants access to every Creative Cloud app. Product-specific licenses (e.g. Photoshop-only) cost less.

Adobe Stock and Acrobat Sign are usage-based (not seat-based), billed on overage. AI generative credits have separate metering.

All Apps vs. single app Usage-based: Stock + Sign AI generative credits

Admin View & Seat Management

The Adobe Admin Console (adminconsole.adobe.com) is the control plane for the entire org. Admin roles: System Admin → Product Admin → Product Profile Admin → User Group Admin. Product Profiles define which apps/services a user can access — they act as a permission template applied to license assignments.

Key limitation: no custom admin roles with granular permissions (all admin roles are fixed). SCIM only on Enterprise + SSO. No last-login tracking, making unused seat identification difficult.

Product profiles as templates No custom admin roles No last-login tracking Storage admin is a role
Design Signal: Adobe's Product Profile system is a strong pattern — pre-configured access bundles that admins assign to users, removing the need to configure per-user. Admin Console docs →

Design Pattern Taxonomy

Six distinct patterns emerged across these platforms. Most platforms blend 2–3 of these. Understanding which patterns your platform already exhibits helps you borrow intelligently from others.

💡
Pattern 1: Unified Seat with Product-Access Flags

One seat per user; the type of that seat determines which products they can access. This simplifies the admin's mental model — there's only one thing to assign per person. Best example: Figma (post-March 2025). The challenge is communicating what each seat type unlocks clearly enough that admins don't get it wrong.

⚠️
Pattern 2: Per-Product Seat Pools

Each product has its own independently managed seat count and plan tier. Users can have different roles across products. Best examples: Monday.com, Atlassian. Powerful but cognitively heavy for admins managing multiple products at once — requires a strong "hub" overview screen to prevent confusion.

✅
Pattern 3: Free Tier + Individual Upgrade

The base product is always free with unlimited seats; specific users can be upgraded to a higher tier with additional capabilities or resources. This is Gloo's own Gloo+ model. The closest analogues are Slack's guest roles and Figma's View seats — both free and unlimited, with upgrade paths for individuals who need more. No platform reviewed does this as the primary tier for a product, making Gloo's model relatively novel.

💡
Pattern 4: License as Permission Profile

A pre-configured access template (profile/bundle) is assigned to a user, granting a consistent set of capabilities without per-user configuration. Best examples: Adobe (Product Profiles), Salesforce (Permission Set Groups). Reduces admin error at scale; ideal when you have large teams needing consistent access.

⚠️
Pattern 5: Nested Admin Delegation

Org Admin → Group/Dept Admin → End User. Each level can manage only the users and settings below them. Examples: Canva (Org → Team Admin), Monday.com (Admin → Dept Owner), Google Workspace (OU-based), Slack (Org → Workspace Admin). Critical for church networks or multi-campus orgs where a central admin can't manage everyone.

✅
Pattern 6: Usage-Based Resource Meters Within Seats

Seat grants access; a separate resource counter (credits, tokens, API calls) governs how much the user can do within that access. Examples: Figma (AI credits, org-pooled), Adobe (generative credits), Canva (Magic Studio usage caps). Gloo needs this for Content Studio (AI credits) and Ministry Chat (AI tokens). The key UX question is where and how these meters are surfaced — to the user, the admin, or both.

Design Recommendations for Gloo Workspace

REC 01

Build a "Workspace Hub" Admin View — Not Just Per-Product Admin

The most effective pattern across Atlassian, Monday, and Adobe is a single org-level dashboard where admins can see all members and all product access states in one place. Each row = a member; columns = each product + their seat status within it. This prevents the "tab-switching" admin experience.

Inspired by: Atlassian admin.atlassian.com · Monday.com Administration panel
REC 02

Use a Self-Service Seat Request Flow with a "Pending" Grace State

Rather than admin-only seat assignment, let members request access to a product or request a Gloo+ upgrade. Admins approve/deny. Figma's 3-day temporary access while pending is a strong model — it reduces friction for the member while protecting the admin's budget control.

Inspired by: Figma seat request + temp access flow
REC 03

Treat Gloo+ as an "Upgrade Overlay" — Not a Separate Tier

Since Gloo Communications is always free with unlimited seats, the Gloo+ upgrade should feel like a capability unlock within a free product — not a plan change. Design it as a toggle/badge on individual members in the admin table: a "+" indicator next to their name. The upgrade path should be inline, not redirect-to-billing.

Inspired by: Figma View → Full seat type change · Notion guest → member upgrade
REC 04

Surface AI Resource Meters at Both Admin and Member Level

For Content Studio (AI credits) and Ministry Chat (AI tokens), show two views: an admin aggregate view (total credits/tokens purchased, consumed, remaining across org + per-member breakdown) and a member personal view (my credits, my usage this month). Consider org-level credit pooling as default with optional per-member caps.

Inspired by: Figma org-pooled AI credits · Adobe generative credit tracking · Canva per-user Magic Studio caps
REC 05

Adopt a Nested Admin Delegation Model for Multi-Campus Use

If Gloo serves multi-campus churches or networks, the Org Admin → Workspace Admin (per campus/church) → Member hierarchy is critical. Use Canva's pattern of "org sets defaults, sub-admins can override within bounds." This prevents central IT becoming a bottleneck while preserving top-level governance.

Inspired by: Canva Enterprise org/team structure · Slack org vs. workspace admin split
REC 06

Design Clear "Product Non-Member" States

When a user has access to the Workspace but hasn't been assigned to a specific product, what do they see? Monday's "Product Non-Member" pattern — read-only access by default — is worth adopting. Members should land in the Workspace, discover all products, and see clear "Request Access" entry points. Avoid empty states or 404-style blocks.

Inspired by: Monday.com Product Non-Member role · Figma View seat as default for new joiners
⚠️
Anti-Pattern to Avoid: Billing-on-Invite

Atlassian charges as soon as a user is invited, even before acceptance. This causes admin anxiety and over-provisioning. Gloo should bill only on acceptance/activation, with a clear "pending invite" state that doesn't incur cost.

05

Proposed Admin User Path

A recommended flow for a Workspace Admin managing their organization. This covers the most common tasks and is designed to minimize context-switching.

01

Workspace Hub — Members Overview

Admin lands on a unified member table. Each row = one member. Columns show: Name, Role, Product Access (icons per product with seat status), Gloo+ status, Last Active.

  • Filter by product, seat status, or role
  • Bulk-select for role changes or product assignments
  • Pending invites shown in a distinct state (no cost yet)
  • Notification badge for pending seat upgrade requests
02

Invite Member → Choose Product Access

Admin invites by email. A second step shows all Workspace products — admin checks which products to grant access to and what seat tier. Gloo Communications is pre-checked as free. AI products show credit/token allocation upfront.

  • Products listed with current seat cost impact
  • Running total of monthly cost change shown
  • Invite sent — member appears as "Pending" (no billing)
  • Billing starts on first login / acceptance
03

Handle Seat Upgrade Requests

Members can request access to a product or a Gloo+ upgrade from within the product. Admin receives a notification queue. Each request shows: who, what product/tier, and the cost impact.

  • One-tap Approve / Deny from notification or request queue
  • Optional: grant temporary access (3-day, inspired by Figma)
  • Denied requests trigger a customizable message to the member
04

Manage a Member's Access (Inline Panel)

Clicking a member opens a right-side panel — no full-page navigation. Shows their complete access state across all products, Gloo+ status, and AI resource usage for this billing period.

  • Seat type change per product — dropdown, immediate
  • Toggle Gloo+ on/off with cost preview
  • AI usage bar: X credits used of Y allocated (Content Studio)
  • AI token usage bar (Ministry Chat) — same pattern
  • Remove from workspace → releases seat at next billing cycle
05

Subscription & Usage Overview

A separate Billing section (not mixed into member management) gives the admin a product-by-product breakdown: seats used vs. purchased, AI credits/tokens consumed vs. available, and the projected next invoice.

  • Per-product cost cards with seat count controls
  • AI resource pools — aggregate + per-member drilldown
  • Upgrade / downgrade product plans from this view
  • Invoice history and next billing date
06

Member's Own View (End-User)

A member sees only their own access state — what products they have, their Gloo+ status, and their personal AI credit/token usage. No access to billing, other members' data, or subscription management (unless they have an admin role).

  • My Products: icons + access level for each
  • Request Access button on locked/unassigned products
  • My AI usage: credits remaining this month (Content Studio)
  • Upgrade to Gloo+ prompt if on free tier (if allowed by admin)
